## 1. 技术栈与项目结构

### 技术选型

- **语言：** TypeScript
- **渲染：** HTML5 Canvas（游戏画面）+ HTML/CSS（UI层覆盖在Canvas上）
- **构建工具：** Vite（快速开发服务器，TypeScript原生支持）
- **目标平台：** PC浏览器（仅支持鼠标操作）
- **数据持久化：** 无（Prototype阶段不做存档，刷新页面重置）

### 项目结构

```
sisyphus-game/
├── index.html
├── package.json
├── tsconfig.json
├── vite.config.ts
├── src/
│   ├── main.ts                 # 入口文件，初始化游戏
│   ├── config.ts               # ★ 所有可配置参数集中定义
│   ├── game/

│   │   ├── ClimbingState.ts    # 攀登阶段逻辑
│   │   ├── ShopState.ts        # 商店阶段逻辑
│   │   ├── JudgmentBar.ts      # 判定条机制
│   │   ├── StaminaSystem.ts    # 耐力系统
│   │   ├── CheckpointSystem.ts # 检查点与奖励
│   │   ├── QTESystem.ts        # QTE机制
```

### 架构说明

- **Canvas层** 负责渲染游戏画面（山、角色、石头、检查点等）
- **HTML/CSS层** 覆盖在Canvas上，负责所有UI元素（判定条、QTE表盘、商店界面、HUD）。这样UI不受镜头缩放影响。
- **配置文件 `config.ts`** 是所有数值的唯一来源（Single Source of Truth），所有模块从此文件读取参数。

## 2. 游戏概述

### 背景设定

玩家扮演西西弗斯，在冥界的山上推石头。山上散落着其他亡魂遗留的冥币（oboloi）。每次攀登注定以石头滚落告终。在山脚下，商人之神赫尔墨斯（Hermes）经营一家商店，用冥币交换各种能力升级和神器。

### 核心循环（单局）

```
山脚出发（耐力满）
    → 按住鼠标发起判定
    → 判定成功：推动石头上升固定距离，消耗耐力
    → 判定失败：无推动，不消耗耐力，但浪费时间触发下滑风险
    → 到达检查点：获得货币奖励（可选触发QTE翻倍）
    → 耐力逐渐消耗 → 成功区域越来越窄
    → 耐力耗尽 → 无法成功判定 → 连续失败 → 冒汗 → 下滑 → 滚落
    → 回到山脚 → 进入商店
```

### 跨局进展（Incremental）

```
多次攀登积累货币
    → 商店购买普通升级（推力、耐力、恢复速度等）
    → 达到高度里程碑获得锭（稀有货币）
    → 铸造神器（解锁新机制：暴击、双手、楔子、QTE）
    → 能推到更高的检查点 → 获得更多货币 → 继续升级
```

### 前期节奏预期

- **第1-2局：** 可能到不了第1个检查点，或刚好到达。
- **第3-5局：** 稳定到达第1个检查点，开始积累货币。
- **第6-10局：** 存够钱升级，可以到达第2个检查点。
- 之后每个新检查点需要更多轮次才能到达。

## 3. 核心循环详解

### 游戏状态机

游戏有两个主要状态：

```
┌─────────────┐     石头滚落      ┌─────────────┐
│  CLIMBING   │ ───────────────→ │    SHOP     │
│  (攀登中)    │                  │  (商店)      │
└─────────────┘ ←─────────────── └─────────────┘
                   点击"出发"
```

**CLIMBING状态：**
- 游戏主循环运行
- 玩家可以进行判定操作
- 耐力系统、下滑系统、检查点系统、楔子系统均活跃
- 当石头滚落到山脚后，切换到SHOP状态

**SHOP状态：**
- 游戏主循环暂停（或只渲染商店画面）
- 显示商店UI，玩家可以购买升级
- 玩家点击"出发"按钮后，以当前升级后的属性开始新一局，切换回CLIMBING状态

### 单局初始化

每局开始时：
- 石头位置 = 0（山脚）
- 当前耐力 = 玩家的耐力上限（由升级决定）
- 楔子库存 = 玩家当前拥有的楔子数量
- 检查点奖励重置（每局重新领取）
- 下滑状态重置

## 4. 判定条机制（主要交互）

### 概述

这是游戏的核心交互。玩家按住鼠标左键，角色头顶出现一个横向判定条，条上有一个指针从左向右匀速移动并来回反弹。玩家在合适的时机松开鼠标，根据指针位置判定成功或失败。

### 详细流程

1. **玩家按下鼠标左键：**
   - 角色头顶出现判定条
   - 指针从判定条最左端开始，以恒定速度向右移动
   
2. **指针移动：**
   - 以 `JUDGMENT_POINTER_SPEED` 的速度匀速移动
   - 到达右端后反弹向左移动，到达左端再反弹向右，持续来回
   
3. **玩家松开鼠标左键：**
   - 读取指针当前位置
   - 判断指针是否在成功区域内
   - **成功：** 石头向上推动 `PUSH_DISTANCE` 的距离，消耗耐力
   - **失败：** 无推动，不消耗耐力，重置下滑计时器（见下滑机制）

4. **判定条消失，等待下一次按住**

### 判定条结构

```
判定条示意图（从左到右）：

[  失败  |  失败  | 成功区域 |  失败  |  失败  ]
                  [暴击2][暴击1]
                  
成功区域居中于判定条。
暴击区域在成功区域的正中心。
二阶暴击区域在暴击区域的正中心。
```

- **判定条总宽度：** `JUDGMENT_BAR_TOTAL_WIDTH`（像素，UI层固定大小，不受镜头缩放影响）
- **成功区域宽度：** 由当前耐力决定（见耐力系统）
- **成功区域位置：** 始终居中于判定条
- **失败区域：** 判定条中除成功区域外的所有部分

### 可配置参数

| 参数名 | 说明 | 建议初始值 |
|--------|------|-----------|
| `JUDGMENT_BAR_TOTAL_WIDTH` | 判定条总宽度（像素） | 300 |
| `JUDGMENT_BAR_HEIGHT` | 判定条高度（像素） | 20 |
| `JUDGMENT_POINTER_SPEED` | 指针移动速度（像素/秒） | 200 |
| `JUDGMENT_SUCCESS_COLOR` | 成功区域颜色 | `#4CAF50` |
| `JUDGMENT_FAIL_COLOR` | 失败区域颜色 | `#f44336` |
| `JUDGMENT_POINTER_COLOR` | 指针颜色 | `#FFFFFF` |
| `JUDGMENT_BAR_Y_OFFSET` | 判定条在角色头顶上方的偏移（像素） | -40 |

## 5. 耐力系统

### 概述

耐力是限制玩家每局上升高度的核心资源。耐力决定判定条上成功区域的宽度。耐力高时成功区域宽，容易判定成功；耐力低时成功区域窄，判定变难；耐力为0时成功区域消失，玩家不可能成功判定。

### 机制细节

- **耐力是隐形的：** 不直接显示耐力数值给玩家。玩家通过成功区域宽度的变化来感知耐力状态。
- **耐力消耗：** 每次**成功判定**消耗 `STAMINA_COST_PER_SUCCESS` 点耐力。**失败判定不消耗耐力。**
- **耐力恢复：** 耐力随时间缓慢恢复，速率为 `STAMINA_REGEN_RATE`（点/秒）。无论玩家是否在操作，耐力都在恢复（但恢复速度极慢）。
- **每局初始化：** 每局开始时耐力 = `STAMINA_MAX`（由升级决定）。

### 成功区域宽度计算

```typescript
// 成功区域宽度 = 判定条总宽度 × 当前耐力比例 × 最大成功区域占比
successZoneWidth = JUDGMENT_BAR_TOTAL_WIDTH 
                   * (currentStamina / STAMINA_MAX) 
                   * SUCCESS_ZONE_MAX_RATIO
```

其中 `SUCCESS_ZONE_MAX_RATIO` 是耐力满时成功区域占判定条总宽度的最大比例。

### 可配置参数

| 参数名 | 说明 | 建议初始值 |
|--------|------|-----------|
| `STAMINA_MAX_BASE` | 基础耐力上限（未升级时） | 100 |
| `STAMINA_COST_PER_SUCCESS` | 每次成功判定消耗的耐力 | 5 |
| `STAMINA_REGEN_RATE_BASE` | 基础耐力恢复速率（点/秒），未升级时 | 0.25 |
| `SUCCESS_ZONE_MAX_RATIO` | 耐力满时成功区域占判定条的最大比例 | 0.6 |
| `SUCCESS_ZONE_MIN_WIDTH` | 成功区域的最小宽度（像素），低于此值视为0 | 5 |

### 设计意图说明

- 耐力恢复极慢（初始约0.25点/秒，即20秒恢复一次判定消耗的5点），意味着玩家不可能通过"等"来无限推进。
- 但恢复机制为skill progression留出空间：高手可以在失败判定之间利用自然恢复的微量耐力多推一两次。
- 楔子系统（见后文）可以让玩家安全地等待耐力恢复，形成策略深度。

## 6. 下滑与失败机制

### 概述

当玩家无法成功推动石头时（连续失败判定或不操作），石头会开始下滑，最终加速滚落到山脚，结束本局。

### 状态机

```
正常状态
    │
    ├── 判定成功 → 重置下滑计时器，回到正常状态
    │
    └── 判定失败 或 不操作
         │
         ├── 经过 SWEAT_DELAY 秒 → 冒汗状态
         │                          （角色额头出现汗珠动画）
         │                          （石头开始微微往下滑）
         │
         └── 再经过 SLIDE_DELAY 秒 → 下滑状态
                                      （不可逆）
                                      （石头加速下滑）
                                      （经过 SLIDE_DURATION 秒后滚落到山脚）
                                      → 本局结束
```

### 关键时间窗口

从最后一次成功判定开始计时：

- **0 ~ SWEAT_DELAY 秒：** 正常状态。玩家可以尝试判定。
- **SWEAT_DELAY ~ (SWEAT_DELAY + SLIDE_DELAY) 秒：** 冒汗状态。角色冒汗，石头开始微微下滑，但玩家仍可尝试判定。如果此时判定成功，回到正常状态。
- **(SWEAT_DELAY + SLIDE_DELAY) 秒之后：** 下滑状态。**不可逆。** 石头加速下滑，玩家无法操作。石头在 `SLIDE_DURATION` 秒内滚落到山脚。

### 总时间窗口

从最后一次成功判定到不可逆下滑，总共约 `SWEAT_DELAY + SLIDE_DELAY` 秒。这个时间大约等于2-3次判定回合（一次判定回合 ≈ 指针来回一次的时间）。也就是说，如果玩家连续失败3-4次判定，就会进入不可逆下滑。

### 冒汗状态下的微滑

在冒汗状态下，石头以 `SWEAT_SLIDE_SPEED` 的速度缓慢下滑。这个下滑是可以被成功判定"推回去"的——但由于耐力已经很低（通常这时候成功区域很窄），推回去很难。

### 滚落动画

进入不可逆下滑后：
- 石头的下滑速度从 `SLIDE_INITIAL_SPEED` 线性加速到 `SLIDE_MAX_SPEED`
- 角色跟着石头一起往下跑/滑
- 石头到达山脚（位置=0）时，播放一个简短的碰撞/停止动画
- 然后切换到SHOP状态

### 可配置参数

| 参数名 | 说明 | 建议初始值 |
|--------|------|-----------|
| `SWEAT_DELAY` | 从最后一次成功判定到开始冒汗的时间（秒） | 1.0 |
| `SLIDE_DELAY` | 从开始冒汗到不可逆下滑的时间（秒） | 3.0 |
| `SLIDE_DURATION` | 不可逆下滑到滚落山脚的时间（秒） | 2.0 |
| `SWEAT_SLIDE_SPEED` | 冒汗状态下的缓慢下滑速度（游戏单位/秒） | 5 |
| `SLIDE_INITIAL_SPEED` | 不可逆下滑的初始速度 | 20 |
| `SLIDE_MAX_SPEED` | 不可逆下滑的最大速度 | 200 |

## 7. 检查点与奖励系统

### 概述

山坡上有固定位置的检查点。玩家每次推石头经过检查点时，获得该检查点标注的货币奖励。每局检查点位置和奖励不变，且每局可以重复领取（每局进度从0开始）。

### 检查点设计

检查点在山坡上的位置是固定的，定义在配置中：

```typescript
// 检查点配置：[距离山脚的高度, 铜币奖励, 银币奖励, 金币奖励]
CHECKPOINTS: [
  { height: 50,   reward: { obolus: 10,  drachma: 0, stater: 0 } },
  { height: 120,  reward: { obolus: 25,  drachma: 0, stater: 0 } },
  { height: 220,  reward: { obolus: 50,  drachma: 0, stater: 0 } },
  { height: 350,  reward: { obolus: 100, drachma: 0, stater: 0 } },
  { height: 520,  reward: { obolus: 0,   drachma: 1, stater: 0 } },
  // ... 更多检查点
]
```

### 检查点间距设计原则

- 低处检查点间距小（50-70），奖励少，玩家前期容易到达前几个
- 越高间距越大（100 → 130 → 170 → ...），奖励也越丰厚
- 间距增长与升级后的能力增长相匹配，确保玩家总是需要多次升级才能到达下一个检查点

### 到达检查点时的表现

- 石头经过检查点位置时，播放一个获得货币的动画（货币图标飘出）
- HUD上显示获得的货币数量
- 如果玩家已解锁QTE神器，此时额外显示一个"挑战"按钮（见QTE系统）

### 高度里程碑（锭奖励）

除了普通检查点外，某些特殊高度是"里程碑"。玩家**历史上第一次**到达该高度时，获得锭（铸造神器的稀有货币）。这是一次性奖励，之后再到达不会重复给。

```typescript
MILESTONES: [
  { height: 120,  ingotReward: 1 },
  { height: 350,  ingotReward: 1 },
  { height: 520,  ingotReward: 2 },
  // ... 更多里程碑
]
```

### 可配置参数

| 参数名 | 说明 | 建议初始值 |
|--------|------|-----------|
| `CHECKPOINTS` | 检查点配置数组（高度 + 各货币奖励） | 见上方示例 |
| `MILESTONES` | 里程碑配置数组（高度 + 锭奖励） | 见上方示例 |
| `CHECKPOINT_VISUAL_RADIUS` | 检查点在画面中的视觉大小 | 15 |
| `CHECKPOINT_COLLECT_ANIMATION_DURATION` | 领取奖励动画时长（秒） | 0.5 |

## 8. 货币系统

### 货币类型

| 货币名 | 内部名称 | 用途 | 获取方式 |
|--------|----------|------|----------|
| 奥波勒斯（铜币） | `obolus` | 购买基础升级 | 低处检查点奖励 |
| 德拉克马（银币） | `drachma` | 购买高级升级 | 高处检查点奖励 |
| 斯塔特尔（金币） | `stater` | 购买顶级升级 | 极高处检查点奖励 |
| 锭 | `ingot` | 铸造神器（特殊升级） | 高度里程碑一次性奖励 |

### 货币换算

```
10000 obolus = 1 drachma
10000 drachma = 1 stater
锭不可换算，只能通过里程碑获取
```

### 是否需要实现货币兑换功能？

Prototype阶段建议**不实现**主动兑换（玩家不能手动把铜换银）。如果升级价格设计得当，玩家自然会在不同阶段获得对应的货币来购买对应等级的升级。后期可以加入兑换功能。

| `OBOLUS_TO_DRACHMA_RATE` | 铜换银的换算比例 | 10000 |
| `DRACHMA_TO_STATER_RATE` | 银换金的换算比例 | 10000 |

## 11. 暴击系统（神器解锁）

### 前置条件

需要装备神器"阿瑞斯的怒火"（`criticalHit`）。

### 机制

装备后，判定条的成功区域中央出现一个小的暴击区域。指针落在暴击区域内时，推动距离大于普通成功。

### 暴击区域规则

- **一阶暴击区域：** 固定宽度 `CRIT_ZONE_WIDTH`，位于成功区域正中央。当成功区域宽度低于 `CRIT_ZONE_MIN_SUCCESS_WIDTH` 时，暴击区域消失。
- **一阶暴击区域升级（普通升级）：** 暴击区域从固定宽度变为成功区域的百分比（`CRIT_ZONE_RATIO`），跟随成功区域缩放。此时暴击区域不会因耐力降低而消失（只是变小）。
- **二阶暴击区域（普通升级）：** 在一阶暴击区域中央再出现一个更小的固定宽度区域 `CRIT2_ZONE_WIDTH`。命中二阶暴击，推动距离更远。二阶暴击区域也在一阶暴击区域足够宽时才显示。

### 判定条结构（带暴击）

```
[  失败  |   成功区域   |  失败  ]
           [ 一阶暴击  ]
            [二阶暴击]
```

判定优先级：二阶暴击 > 一阶暴击 > 普通成功 > 失败

### 暴击相关普通升级

这些升级需要先铸造并装备"阿瑞斯的怒火"后才在商店中可见：

| 升级名 | 效果 | 前置 |
|--------|------|------|
| 怒火淬炼 | 增大暴击推动距离倍率 | 装备criticalHit |
| 精准怒火 | 暴击区域从固定宽度变为成功区域百分比缩放 | 怒火淬炼 Lv3 |
| 二阶暴击 | 解锁二阶暴击区域 | 精准怒火 Lv5 |

### 可配置参数

| 参数名 | 说明 | 建议初始值 |
|--------|------|-----------|
| `CRIT_ZONE_WIDTH` | 一阶暴击区域固定宽度（像素） | 20 |
| `CRIT_ZONE_RATIO` | 升级后，一阶暴击区域占成功区域的比例 | 0.3 |
| `CRIT_ZONE_MIN_SUCCESS_WIDTH` | 成功区域低于此宽度时暴击消失（像素） | 30 |
| `CRIT_MULTIPLIER_BASE` | 一阶暴击的推力倍率（相对普通推动） | 2.0 |
| `CRIT2_ZONE_WIDTH` | 二阶暴击区域固定宽度（像素） | 8 |
| `CRIT2_MULTIPLIER_BASE` | 二阶暴击的推力倍率 | 4.0 |

## 12. 双手推石系统（神器解锁）

### 前置条件

需要装备神器"赫拉克勒斯的臂环"（`dualPush`）。

### 机制

装备后，玩家可以交替使用左键和右键进行判定。左键和右键各有独立的耐力条。

**操作流程：**
1. 按住左键 → 左手判定条出现 → 松开左键判定（消耗左手耐力）
2. 按住右键 → 右手判定条出现 → 松开右键判定（消耗右手耐力）
3. 交替进行

**限制：** 不能同时按住两个键。如果同时按住，只响应先按下的那个键。

### 耐力独立

- 左手耐力和右手耐力各自独立计算，各自消耗和恢复
- 初始值都等于 `STAMINA_MAX`
- 等效于总耐力翻倍，但需要玩家协调两只手

### UI表现

- 左手判定条和右手判定条可以用不同颜色区分
- 或者只显示一个判定条，但左键和右键的成功区域宽度（由各自耐力决定）会不同

### 可配置参数

| 参数名 | 说明 | 建议初始值 |
|--------|------|-----------|
| `DUAL_PUSH_LEFT_COLOR` | 左手判定条颜色 | `#4CAF50` |
| `DUAL_PUSH_RIGHT_COLOR` | 右手判定条颜色 | `#2196F3` |

## 13. 楔子系统（神器解锁）

### 前置条件

需要装备神器"赫菲斯托斯的楔子"（`wedge`）。

### 机制

楔子是消耗品。玩家在攀登过程中可以使用楔子将石头固定在当前位置。使用后，石头不会因为不操作或失败判定而下滑，玩家可以安全地等待耐力恢复。

### 使用方式

- **触发：** 玩家按下特定按键（例如空格键 `WEDGE_KEY`）使用一个楔子
- **效果：** 石头被固定，下滑机制暂停。持续 `WEDGE_DURATION` 秒后楔子断裂，恢复正常状态
- **限制：** 楔子断裂时有冷却时间，不能立即使用下一个
- **策略：** 在耐力中段使用楔子，等待恢复，然后继续推动。最大化利用恢复窗口。

### 楔子获取

楔子在商店中用普通货币购买，每局携带的楔子数量有上限。

### 楔子相关普通升级

| 楔子背包 | 增加每局携带楔子上限 | 装备wedge |

### 可配置参数

| 参数名 | 说明 | 建议初始值 |
|--------|------|-----------|
| `WEDGE_KEY` | 使用楔子的按键 | `Space` |
| `WEDGE_DURATION_BASE` | 楔子持续时间（秒） | 5.0 |
| `WEDGE_COOLDOWN` | 楔子断裂后的冷却时间（秒） | 2.0 |

## 14. QTE系统（神器解锁）

### 前置条件

需要装备神器"命运之轮"（`qte`）。

### 触发

当玩家到达一个检查点时，如果装备了QTE神器，检查点旁会额外显示一个"挑战命运"按钮。玩家可以选择：
- **不触发：** 正常领取检查点奖励，继续攀登
- **触发QTE：** 进入QTE挑战界面

### QTE界面：命运之轮

一个圆形表盘，中心有一根指针以恒定速度顺时针旋转。表盘被分成若干扇区，分为两种：

- **可点击区域（绿色/亮色）：** 玩家应该在此区间疯狂点击
- **禁止区域（红色/暗色）：** 玩家不能点击

### QTE规则

- **时限：** QTE持续 `QTE_DURATION` 秒
- **目标：** 在可点击区域内累积足够的点击次数，达到 `QTE_TARGET_CLICKS` 次即成功
- **惩罚：** 在禁止区域点击，累积 `QTE_PENALTY_LIMIT` 次后**立即失败**
- **结果：**
  - **成功：** 该检查点奖励翻 `QTE_REWARD_MULTIPLIER` 倍
  - **失败：** 直接滑落到山脚，本局结束（且不获得该检查点奖励）

### 表盘配置

```typescript
QTE_WHEEL_SEGMENTS: [
  { type: "safe",   angleDeg: 60 },   // 可点击，60度
  { type: "danger", angleDeg: 30 },   // 禁止，30度
  { type: "safe",   angleDeg: 45 },
  { type: "danger", angleDeg: 45 },
  { type: "safe",   angleDeg: 90 },
  { type: "danger", angleDeg: 30 },
  { type: "safe",   angleDeg: 60 },
  // 总和 = 360度
]
```

### 视觉风格

- 圆形表盘做成古希腊风格（石质纹理、金色边框）
- 指针做成命运之针的感觉
- 可点击区域用温暖的金色/绿色
- 禁止区域用暗红/黑色

### QTE相关普通升级

| 升级名 | 效果 | 前置 |
|--------|------|------|
| 命运眷顾 | 增加QTE奖励倍率 | 装备qte |
| 命运容错 | 增加禁止区域的容错次数 | 装备qte |

### 可配置参数

| 参数名 | 说明 | 建议初始值 |
|--------|------|-----------|
| `QTE_DURATION` | QTE持续时间（秒） | 8.0 |
| `QTE_TARGET_CLICKS` | 成功所需的点击次数 | 30 |
| `QTE_PENALTY_LIMIT` | 禁止区域容错点击次数 | 3 |
| `QTE_REWARD_MULTIPLIER_BASE` | 成功后奖励倍率 | 2.0 |
| `QTE_POINTER_SPEED` | 表盘指针旋转速度（度/秒） | 180 |
| `QTE_WHEEL_SEGMENTS` | 表盘扇区配置数组 | 见上方示例 |
| `QTE_WHEEL_RADIUS` | 表盘半径（像素） | 120 |
| `QTE_SAFE_COLOR` | 可点击区域颜色 | `#C8A96E` |
| `QTE_DANGER_COLOR` | 禁止区域颜色 | `#8B0000` |

## 15. 画面表现与镜头系统

### 整体美术风格

- **程序生成画面**，不使用外部图片资源（Prototype阶段）
- **角色：** 黑色抽象火柴人剪影
- **石头：** 灰色/深色抽象圆形
- **山体：** 从左下到右上的斜坡，颜色偏拟真（土黄、岩石灰、随高度变化）
- **背景：** 深色（冥界氛围），可以有简单的视差效果
- **检查点：** 山坡上的发光标记点
- **整体色调：** 偏暗、沉稳，但检查点和奖励用亮色突出

### 山体渲染

山体是一个从左到右、从低到高的斜坡。简化为一条斜线或缓曲线，角色和石头沿着斜坡表面移动。

```
画面示意（初始状态，人物较大）：

         ╱ (山继续延伸)
        ╱
       ╱  ★ 检查点
      ╱
     ╱
    ╱  ○  ← 石头
   ╱  ♂  ← 西西弗斯
  ╱________
  山脚
```

### 镜头系统

- **跟随：** 镜头始终跟随角色/石头，保持其在画面中的可视位置
- **缩放：** 随着高度增加，镜头逐渐zoom out，画面中容纳的距离范围增大，人物相对变小
- **缩放模式：** 先实现连续缩放（基于当前高度线性/对数缩放），也预留阶段性缩放的配置选项，后续对比测试
- **UI不受缩放影响：** 判定条、QTE表盘、HUD等所有UI元素在HTML/CSS层，不受Canvas缩放影响

### 缩放计算

```typescript
// 连续缩放模式
zoom = CAMERA_BASE_ZOOM / (1 + currentHeight * CAMERA_ZOOM_RATE)

// 阶段性缩放模式（备选）
zoomLevel = Math.floor(currentHeight / CAMERA_ZOOM_STEP_HEIGHT)
zoom = CAMERA_BASE_ZOOM / (1 + zoomLevel * CAMERA_ZOOM_STEP_FACTOR)
```

### 滚落动画时的镜头

石头滚落时，镜头快速跟随下滑，zoom也快速回到初始值。

### 可配置参数

| 参数名 | 说明 | 建议初始值 |
|--------|------|-----------|
| `CAMERA_BASE_ZOOM` | 初始缩放级别 | 1.0 |
| `CAMERA_ZOOM_RATE` | 连续缩放时，每单位高度的缩放系数 | 0.002 |
| `CAMERA_ZOOM_STEP_HEIGHT` | 阶段性缩放时，每隔多少高度zoom out一级 | 100 |
| `CAMERA_ZOOM_STEP_FACTOR` | 阶段性缩放时，每级缩放的幅度 | 0.3 |
| `CAMERA_ZOOM_MODE` | 缩放模式选择（"continuous" 或 "stepped"） | "continuous" |
| `CAMERA_FOLLOW_SPEED` | 镜头跟随平滑速度 | 5.0 |
| `MOUNTAIN_SLOPE_ANGLE` | 山坡角度（度） | 30 |
| `MOUNTAIN_BASE_COLOR` | 山体底部颜色 | `#8B7355` |
| `MOUNTAIN_TOP_COLOR` | 山体高处颜色 | `#6B6B6B` |
| `BACKGROUND_COLOR` | 背景颜色 | `#1a1a2e` |

```typescript
interface PersistentState {

  // 统计
  totalRuns: number;             // 总局数
```

## 19. 分块开发计划

将开发拆分为多个可独立验收的阶段。每个阶段完成后应能运行并展示该阶段的功能。

---

### 阶段 1：项目骨架与基础渲染 (Block 1)

**目标：** 能看到一个山坡、一个火柴人和一个石头，镜头能跟随。

**任务：**
1. 初始化Vite + TypeScript项目
2. 创建HTML页面，包含全屏Canvas
3. 实现 `config.ts`，填入所有山体和画面相关参数
4. 实现 `Renderer.ts`：Canvas初始化、每帧清屏
5. 实现 `MountainRenderer.ts`：根据角度绘制斜坡
6. 实现 `CharacterRenderer.ts`：在斜坡上绘制火柴人 + 石头
7. 实现 `Camera.ts`：镜头跟随和缩放（先实现连续模式）
8. 用一个临时变量模拟高度变化，验证镜头缩放效果

**验收标准：**
- 打开浏览器可以看到山坡、火柴人、石头
- 修改高度变量时，画面正确滚动和缩放
- 修改config中的颜色、角度等参数时，画面相应变化

---

### 阶段 2：判定条核心交互 (Block 2)

**目标：** 玩家可以按住鼠标、看到判定条、松开鼠标进行判定，成功时石头移动。

**任务：**
1. 实现 `JudgmentBar.ts`：判定条逻辑（指针移动、区域判定）
2. 实现 `JudgmentBarUI.ts`：HTML/CSS覆盖层绘制判定条
3. 实现鼠标事件监听（mousedown → 显示判定条 + 指针开始移动，mouseup → 判定）
4. 成功判定时调用推动逻辑，石头上移 `PUSH_DISTANCE_BASE`
5. 判定条位置跟随角色头顶（但不受Canvas缩放影响）

**验收标准：**
- 按住左键出现判定条，指针来回移动
- 松开左键，根据指针位置输出"成功"或"失败"到控制台
- 成功时石头向上移动，画面滚动
- 修改指针速度、判定条宽度等参数时效果正确变化

---

### 阶段 3：耐力系统 (Block 3)

**目标：** 成功判定消耗耐力，成功区域逐渐缩小，耐力极缓慢恢复。

**任务：**
1. 实现 `StaminaSystem.ts`
2. 成功判定扣耐力，失败不扣
3. 根据当前耐力计算成功区域宽度
4. 时间流逝恢复耐力（极慢）
5. 在判定条UI上正确显示变化的成功区域宽度
6. （可选）添加调试用耐力显示（开发时用，发布时隐藏）

**验收标准：**
- 连续成功判定后，成功区域明显缩小
- 等待一段时间后，成功区域略微恢复
- 耐力耗尽时成功区域消失，无法再判定成功
- 修改耐力参数时效果正确变化

---

### 阶段 4：下滑与失败机制 (Block 4)

**目标：** 无法成功时触发冒汗→下滑→滚落→回到山脚。

**任务：**
1. 实现 `SlideSystem.ts`：下滑状态机
2. 实现冒汗视觉效果（角色头顶汗珠）
3. 实现冒汗时的缓慢下滑
4. 实现不可逆下滑的加速动画
5. 石头到达山脚后触发状态切换（先切到一个"局结束"的占位画面）
6. 实现"成功判定重置下滑计时器"的逻辑

**验收标准：**
- 耐力耗尽后连续失败，1秒后角色冒汗
- 再过几秒，石头开始不可逆加速下滑
- 石头滚回山脚，游戏显示"局结束"
- 在冒汗阶段如果成功判定（手动设大成功区域测试），回到正常状态

---

### 阶段 5：检查点与货币 (Block 5)

**目标：** 山上有检查点，推到获得货币。

**任务：**

5. 实现 `HUD.ts`：显示货币数量、当前高度

**验收标准：**
- 山坡上可以看到检查点标记
- 推到检查点时货币增加，有视觉反馈
- HUD正确显示当前货币和高度

## 附录：关键设计决策备忘

1. **耐力是隐形的** — 不给玩家看数字，通过成功区域宽度间接感知
2. **失败判定不消耗耐力** — 这很重要，否则会形成"越失败越难成功"的负反馈螺旋
3. **每局从满耐力开始** — 每局是独立的，跨局进步体现在升级上
4. **所有新机制通过神器解锁** — 一开始游戏极简（只有按住-松开的判定），复杂度随进度解锁
5. **楔子是神器，不是默认功能** — 需要铸造才有
6. **锭只能通过里程碑获得，且只有一次** — 确保神器获取是重要的进度节点
7. **判定条指针速度恒定** — 难度完全由成功区域宽度（耐力）控制，保持简洁
8. **镜头缩放不影响UI** — UI在HTML层，Canvas只渲染游戏世界
